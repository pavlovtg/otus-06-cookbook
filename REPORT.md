# REPORT

## AI-инструменты

### Cline + Claude Sonnet 4

Cline — AI-агент в VS Code на базе модели Claude Sonnet 4 (Anthropic). Использовался для генерации и редактирования документации, создания архитектурных артефактов (ADR, стандарты, AR), написания промтов и workflows. Работает в режиме Plan/Act с доступом к файловой системе и CLI.

### OpenRouter

OpenRouter — прокси-сервис для доступа к языковым моделям. Использовался как провайдер для Cline: позволил подключить Claude Sonnet 4 без прямой подписки на Anthropic API.

### OpenSpec с расширенной схемой ADR

OpenSpec — фреймворк для spec-driven разработки. Использовалась модифицированная схема `spec-driven-with-adr`, расширяющая стандартный OpenSpec: каждый change включает ADR-артефакт для фиксации архитектурных решений наряду с proposal, design, spec и tasks.

### Claude Opus для архитектурных задач

Claude Opus заметно сильнее Claude Sonnet на архитектурных задачах: справляется с задачами, на которых Sonnet ходит по кругу. Для проектирования и рефакторинга архитектурных артефактов имеет смысл переключаться на Opus, несмотря на более высокую стоимость.

## Проблемы и решения


### Выбор структуры хранения архитектурных требований (2026-06-06)

Потребовалось значительное время на поиск подхода, совмещающего несколько требований одновременно: использование OpenSpec как фреймворка для spec-driven разработки, хранение архитектурных решений в ADR и стандартах, минимизация потребления токенов агентом, единая точка входа через `ARCHITECTURE.md`.
Решение: установлено расширение OpenSpec, написаны новые скилы для работы с архитектурой (`adr`, `ar`, `update-architecture-from-adr`, `update-architecture-from-ar`, `update-architecture-from-standard`), доработаны скилы OpenSpec (`openspec-propose`, `openspec-apply-change` и др.) под схему `spec-driven-with-adr`. Итоговая эффективность подхода будет проверена в ходе разработки.

### Skills не обеспечивали консистентность архитектурных артефактов (2026-06-06)

Первоначальная схема skills (`adr`, `ar`, `update-architecture-from-adr`, `update-architecture-from-ar`, `update-architecture-from-standard`) была построена на разделении «низкоуровневого» (создаёт артефакт) и «оркестрирующего» (обновляет связанные файлы и перекрёстные ссылки) skill. На практике связанные файлы оставались рассинхронизированы: артефакт создавался, а индекс, обзор архитектуры и перекрёстные ссылки — нет.

Корневая причина — непонимание момента активации skill. Инструкции skill не входят в базовый контекст и подгружаются только при явном вызове по триггеру. «Оркестрирующий» skill попросту не активировался, и правила обновления связанных файлов не применялись.

Решение: симметричная структура «один артефакт — один самодостаточный skill». Все правила (домен, нумерация, консистентность связанных файлов) живут внутри каждого skill; дублирование между skills допустимо, оно не тратит токены базового контекста.

### Агент игнорирует Plan mode (2026-06-06)

Cline периодически нарушает контракт Plan mode: вместо вывода плана через `plan_mode_respond` начинает выполнять действия. В блоке Thinking при этом явно пишет, что видит read-only режим и «ищет способ выполнить задачу несмотря на ограничение».
Причина — слабая блокировка инструментов: модели доступны read-only средства (`read_file`, `search_files`, `list_files`), и она трактует задачу как «исследование», постепенно соскальзывая в исполнение.
Надёжного обхода нет. Снижает риск: явная фраза в начале промта «сначала составь план, ничего не делай до моего подтверждения», короткие шаги, своевременное прерывание при первых признаках «исполнения вместо планирования».

### Избыточные шаблоны и перекрёстные ссылки в архитектурной документации (2026-06-07)

Изначально созданные шаблоны ADR, AR и стандартов содержали большое количество избыточного текста и обязательных перекрёстных ссылок между документами. Каждое изменение одного артефакта требовало каскадного обновления связанных файлов, что создавало постоянный шум и рассинхронизацию.

Решение: переключение на Claude Opus, который оптимизировал архитектурную документацию — упростил шаблоны, убрал перекрёстные ссылки, смержил дублирующие документы. Навигация между артефактами централизована в `ARCHITECTURE.md`.

### Архитектурное проектирование с агентом до начала реализации (2026-06-07)

Выбор архитектурных решений до начала реализации плохо ложится в работу с агентом. Плюс он постоянно пытается воплотить придуманные решения в коде, вместо того чтобы их проектировать.

Решение:

- Специализированный промт, явно ограничивающий действия агента на этапе проектирования («только анализ и предложения, никакой реализации»). Указание на порядок сбора стребований и согласования NFR, прежде чем что-либо предложить в качестве решения.
- Переключение на более сильную модель — слабые модели хуже удерживают контекст ограничений и чаще соскальзывают в реализацию.

## Удачные шаги

### Архитектурная документация помогает агенту фиксировать решения (2026-06-07)

Наличие обширной системы ведения архитектурной документации (ADR, AR, стандарты, OpenSpec) оказалось удачным решением: агент самостоятельно соображает, как зафиксировать архитектурные решения, принятые в процессе проектирования, не требуя явных инструкций по каждому артефакту.
