## Определение домена

- Конкретный сервис → имя сервиса (например, `order-service`).
- Несколько сервисов → группирующий bounded context, application или `backend`/`frontend`.
- Технология → название технологии (например, `dotnet`, `postgresql`).
- Общие решения → `general`.
- При сомнении — спросить у пользователя.

## Нумерация

- Монотонная в рамках всего репозитория, не повторяется.
- Следующий номер = максимальный существующий + 1 (проверить все папки `docs/adr/`, включая `archive/`).

## Создание ADR

1. Определить домен (правила выше).
2. Определить следующий номер.
3. Создать файл `docs/adr/<domain>/ADR-NNNN-<kebab-name>.md` по шаблону из `assets/adr-template.md`.
4. Обновить индекс ADR в `ARCHITECTURE.md` (добавить строку в таблицу).

## Архивирование ADR

Когда ADR устаревает или заменяется новым:

1. Обновить поле `**Статус**` в файле ADR: `устарел` или `заменён ADR-NNNN`.
2. Переместить файл из `docs/adr/<domain>/` в `docs/adr/archive/`.
3. Обновить индекс ADR в `ARCHITECTURE.md` (обновить статус и путь к файлу).

## Иммутабельность

ADR иммутабельны после принятия. Для изменения решения — создать новый ADR с полем `**Заменяет**`.
