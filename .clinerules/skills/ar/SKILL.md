## Определение домена

- Конкретный сервис → имя сервиса (например, `order-service`).
- Несколько сервисов → группирующий bounded context, application или `backend`/`frontend`.
- Технология → название технологии (например, `dotnet`, `postgresql`).
- Общие правила → `general`.
- При сомнении — спросить у пользователя.

## Нумерация

- Монотонная в рамках всего репозитория, не повторяется.
- Следующий номер = максимальный существующий + 1 (проверить все папки `docs/architecture/rules/`).

## Создание AR

1. Определить домен (правила выше).
2. Определить следующий номер.
3. Создать файл `docs/architecture/rules/<domain>/AR-NNNN-<kebab-name>.md` по шаблону из `assets/ar-template.md`.
4. Обновить индекс AR в `ARCHITECTURE.md` (добавить строку в таблицу).

## Удаление AR

Когда AR больше не актуально:

1. Удалить файл `docs/architecture/rules/<domain>/AR-NNNN-<kebab-name>.md`.
2. Удалить строку из таблицы AR в `ARCHITECTURE.md`.
3. Обновить ссылки в связанных стандартах (убрать ссылку на удалённый AR).

## Принципы

- Каждый AR — небольшой, независимый, легко читаемый документ.
- AR — источник истины для архитектурных ограничений.
- Устаревшие AR удаляются (не архивируются).
