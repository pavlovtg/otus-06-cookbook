# ADR-0012: Domain-Driven Design для доменной логики backend

- **Статус**: принят
- **Домен**: backend
- **Дата**: 2026-06-07

## Контекст

Доменная область проекта (рецепты, ингредиенты, планы меню, пользователи и их взаимодействия) содержит нетривиальные бизнес-инварианты и связи.
Без явного подхода к моделированию домен быстро вырождается в анемичную модель с бизнес-логикой, размазанной по контроллерам и сервисам, что затрудняет сопровождение и эволюцию.

## Рассмотренные варианты

- **Domain-Driven Design (DDD)** — тактические паттерны (Aggregate, Entity, Value Object, Domain Event, Repository, Domain Service) и стратегические (Bounded Context, Ubiquitous Language). Подходит для сложных доменов, требует дисциплины.
- **Transaction Script** — простой подход «процедура на сценарий». Быстр на старте, но не масштабируется на сложные инварианты.
- **Anemic Domain Model** — модели как DTO + сервисы с логикой. Распространённый антипаттерн при DDD-целях.

## Решение

Доменная логика backend-сервисов реализуется по подходу **Domain-Driven Design**:

- бизнес-правила и инварианты живут внутри агрегатов;
- состояние агрегатов изменяется только через их методы (запрет анемичных моделей);
- идентификаторы — типизированные (Value Object), а не «голые» примитивы;
- межагрегатные взаимодействия — через доменные события и application-сервисы;
- границы bounded context соответствуют отдельным сервисам / приложениям в монорепо.

## Последствия

- Требуется единый язык (Ubiquitous Language) между разработчиками и продуктом.
- Повышается порог входа: разработчики должны владеть тактическими паттернами DDD.
- Доменный слой изолирован от инфраструктуры — сочетается с гексагональной архитектурой ([ADR-0013](ADR-0013-hexagonal-architecture.md)).
- Требуются дополнительные тесты доменной логики (unit-тесты агрегатов и VO).
