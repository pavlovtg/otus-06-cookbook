# ADR-0005: JWT-аутентификация

- **Статус**: принят
- **Домен**: general
- **Дата**: 2026-06-06

## Контекст

PRD требует регистрации и авторизации пользователей, разграничения доступа к рецептам (публичные / приватные) и идентификации автора комментариев.
Необходим механизм аутентификации, совместимый с REST API и Docker Compose окружением без внешних сервисов.

## Рассмотренные варианты

- **JWT (JSON Web Token)** — stateless-токены, не требуют хранилища сессий; подписываются секретным ключом; стандарт для REST API.
- **Session-based аутентификация** — сессии хранятся на сервере (в памяти или Redis); требует дополнительного сервиса или sticky sessions.
- **OAuth2 / внешний провайдер** — делегирование аутентификации (Google, GitHub); избыточно для учебного проекта без внешних зависимостей.

## Решение

Выбран JWT для аутентификации и авторизации.

Access-токен передаётся в заголовке `Authorization: Bearer <token>`.
Секретный ключ задаётся через переменную окружения.

## Последствия

- Stateless-архитектура: сервер не хранит состояние сессий.
- Инвалидация токенов до истечения срока требует дополнительного механизма (blacklist) — в MVP не реализуется.
- Два предзаполненных пользователя создаются через seed-данные.
- Стандарт реализации: [docs/standards/api-design.md](../../standards/api-design.md) (заголовок `Authorization: Bearer`).
