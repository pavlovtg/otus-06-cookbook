# ADR-0030: Стратегия CI

Дата: 2026-06-09

## Контекст

Монорепо содержит несколько независимых компонентов (.NET-сервисы, Next.js, Python e2e тесты).
При push в feature-ветку запускать полный прогон всех компонентов избыточно и замедляет обратную связь.
При слиянии в `main` необходима полная проверка для защиты стабильной ветки.
Альтернативы: единый workflow без фильтров; матрица по компонентам.

## Решение

Два режима CI:

1. **Push в feature-ветку** — умный запуск: проверяются только компоненты, затронутые изменёнными файлами.
   e2e и ui-тесты запускаются только при изменении исполняемого кода или конфигурации развёртывания.
2. **PR в `main`** — полный прогон всех проверок без фильтров.
   Падение любой проверки блокирует merge через branch protection rules.

Типы проверок в pipeline: lint, test, build, e2e (API-уровень), ui-test (браузерный).
Независимые проверки выполняются параллельно; build зависит от lint и test своего компонента; e2e и ui-test зависят от всех build.

## Последствия

- Быстрая обратная связь при разработке отдельного компонента.
- Полная защита `main` от регрессий.
- Усложнение конфигурации CI: два workflow-файла и логика определения затронутых компонентов.
