# ADR-0009: Swagger UI публикуется на API Gateway

- **Статус**: принят
- **Домен**: rest-api
- **Дата**: 2026-06-07

## Контекст

ADR-0006 фиксирует REST API как протокол, AR-0002 требует живой Swagger UI в работающем приложении. ADR-0008 вводит API Gateway как единую точку доступа: backend-сервисы становятся внутренними и не публикуются наружу. Внешним потребителям и frontend нужна единая, доступная снаружи точка получения OpenAPI-документации, согласованная с фактической поверхностью API, открытой через gateway.

## Рассмотренные варианты

- **Swagger UI на каждом backend-сервисе, доступный наружу** — конфликтует с ADR-0008 (backend не публикуется); множество разрозненных URL для клиентов.
- **Внешний портал документации (отдельный сайт/сервис)** — лишний компонент; рассинхрон с фактическим API; дополнительный процесс публикации.
- **Swagger UI на API Gateway** — публикуется через ту же точку входа, что и сам API; единый URL для клиентов; согласован с поверхностью, реально выставленной gateway.

## Решение

Swagger UI публикуется на API Gateway и доступен внешним потребителям через ту же точку входа, что и REST API.

- API Gateway — каноничный источник OpenAPI-документации для клиентов системы.
- Backend-сервисы могут предоставлять собственные OpenAPI-документы для внутренних нужд (разработка, тесты), но наружу публикуется только Swagger UI на gateway.
- Способ формирования содержимого Swagger UI на gateway (единый документ или агрегация спецификаций backend-сервисов) намеренно не определяется в данном ADR и будет зафиксирован отдельным решением.

## Последствия

- AR-0002 (живой Swagger UI) реализуется через gateway.
- URL Swagger UI у клиента стабилен и не зависит от количества backend-сервисов.
- Требуется отдельное архитектурное решение по способу сборки содержимого Swagger UI (вне скоупа данного ADR).
