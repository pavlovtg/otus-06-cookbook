# ADR-0008: API Gateway как единая точка доступа

- **Статус**: принят
- **Домен**: rest-api
- **Дата**: 2026-06-07

## Контекст

Архитектура предполагает несколько backend-сервисов и отдельный frontend. Прямое обращение клиентов к каждому сервису увеличивает связанность, размазывает cross-cutting concerns (CORS, авторизация, rate limiting, наблюдаемость) и усложняет эволюцию backend без изменения клиентов. Требуется единая точка доступа, изолирующая backend от внешних потребителей (frontend, внешние клиенты, инструменты разработки).

## Рассмотренные варианты

- **Прямой доступ frontend → каждый backend-сервис** — минимум инфраструктуры; cross-cutting концерны дублируются в каждом сервисе; границы backend утекают в клиентский код.
- **Backend-for-Frontend (BFF) в каждом сервисе** — каждый сервис сам публикует адаптированный API; дублирование политик и кода адаптации; нет единой точки контроля.
- **API Gateway как единая точка доступа** — единая граница, централизованные cross-cutting концерны, backend остаётся внутренней деталью реализации.

## Решение

Вводится API Gateway как единая точка доступа ко всем backend-сервисам.

- Все клиенты (frontend, внешние потребители, инструменты) обращаются к backend только через gateway.
- Backend-сервисы рассматриваются как внутренние компоненты системы и не публикуются напрямую наружу.
- Gateway отвечает за маршрутизацию, авторизацию на входе, CORS, rate limiting, наблюдаемость и иные cross-cutting задачи. Бизнес-логика на gateway не размещается.

## Последствия

- Появляется новый компонент в архитектуре — API Gateway.
- Backend-сервисы становятся внутренними; их публичные адреса и порты не отдаются наружу.
- Cross-cutting политики централизуются на gateway, устраняя дублирование в backend.
