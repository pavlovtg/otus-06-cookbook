# ADR-0008: API Gateway как единая точка доступа

- **Статус**: принят
- **Дата**: 2026-06-07

## Контекст

Архитектура предполагает несколько backend-сервисов и отдельный frontend. Прямое обращение клиентов к каждому сервису увеличивает связанность, размазывает cross-cutting concerns (CORS, авторизация, rate limiting, наблюдаемость) и усложняет эволюцию backend без изменения клиентов. Требуется единая точка доступа, изолирующая backend от внешних потребителей. Альтернативы: прямой доступ frontend→каждый backend (cross-cutting дублируется, границы backend утекают в клиент), BFF в каждом сервисе (дублирование политик, нет единой точки контроля) — отклонены.

## Решение

Вводится API Gateway как единая точка доступа ко всем backend-сервисам.

- Все клиенты (frontend, внешние потребители, инструменты) обращаются к backend только через gateway.
- Backend-сервисы — внутренние компоненты системы, не публикуются напрямую наружу.
- Gateway отвечает за маршрутизацию, авторизацию на входе, CORS, rate limiting, наблюдаемость и иные cross-cutting задачи. Бизнес-логика на gateway не размещается.

## Последствия

- Появляется новый компонент в архитектуре — API Gateway.
- Backend-сервисы становятся внутренними; их публичные адреса и порты не отдаются наружу.
- Cross-cutting политики централизуются на gateway, устраняя дублирование в backend.
