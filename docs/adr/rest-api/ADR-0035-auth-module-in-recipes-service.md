# ADR-0035: Auth как модуль внутри recipes-сервиса

- **Статус**: принят
- **Дата**: 2026-06-16
- **Заменяет**: ADR-0021, ADR-0022

## Контекст

ADR-0021 предполагал выделенный `auth-service` с отдельной БД и S2S-аутентификацией (ADR-0022). В MVP реализован единственный доменный сервис `recipes`; межсервисных вызовов нет, поэтому S2S не нужен. Реализация отдельного сервиса требует значительных затрат (новый .NET-проект, Dockerfile, docker-compose, миграции, тесты) при нулевой архитектурной выгоде. Альтернативы: выделенный auth-service (ADR-0021, отклонён — over-engineering для MVP с одним сервисом), auth в API Gateway (нарушает AR-0004 — gateway без бизнес-логики).

## Решение

Регистрация, логин и управление пользователями реализуются как модуль внутри `recipes`-сервиса. Пользователи хранятся в схеме `cookbook` той же PostgreSQL. JWT выпускается и валидируется `recipes`-сервисом. YARP остаётся чистым прокси без JWT-валидации.

## Последствия

- Нет отдельного контейнера `authorization` в Docker Compose; docker-compose не усложняется.
- S2S (ADR-0022) не нужен и не реализуется.
- `recipes`-сервис становится единственным issuer JWT; AR-0012 и AR-0013 пересматриваются.
- При появлении второго backend-сервиса потребуется вынести auth в отдельный сервис.
