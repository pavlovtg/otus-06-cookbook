# ADR-0021: Auth-service как отдельный сервис

- **Статус**: заменён ADR-0035
- **Дата**: 2026-06-07

## Контекст

ADR-0005 фиксирует JWT-аутентификацию, но не определяет, где живёт issuer токенов. В горизонте 1 год планируется 2–3 backend-сервиса за YARP (ADR-0010); размещение `/auth/*` внутри одного из доменных сервисов нарушает SRP и связывает доступность логина с доступностью домена. Альтернативы: модуль auth внутри доменного сервиса (нарушает SRP, каскадные отказы), внешний IdP (Keycloak) — отклонены как vendor lock-in и over-engineering для MVP.

## Решение

Выдача и обновление JWT выполняется отдельным сервисом `auth-service`. Сервис разворачивается контейнером в Docker Compose, маршрутизируется через YARP (`/auth/*`), хранит данные в PostgreSQL.

## Последствия

- Добавление новых backend-сервисов не требует изменений в auth-service.
- Отказ доменного сервиса не блокирует логин.
- Появляется отдельный bounded context `Identity` со своей моделью данных и миграциями.
