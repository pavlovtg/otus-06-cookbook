# ADR-0022: S2S-аутентификация через OAuth 2.0 client_credentials

- **Статус**: заменён ADR-0035
- **Дата**: 2026-06-07

## Контекст

Между backend-сервисами требуется аутентифицированный вызов (UC4): один сервис обращается к API другого. JWT-аутентификация (ADR-0005) и выделенный auth-service (ADR-0021) уже определены для пользовательских токенов, но способ получения токена для сервисов отдельно не зафиксирован. Альтернативы: статический сервисный JWT в конфиге (нет ротации, выкидывание клиента требует обновления всех сервисов), mTLS внутри Compose (избыточно, не покрывает claims) — отклонены.

## Решение

Сервис-клиенты получают JWT через OAuth 2.0 grant `client_credentials` на эндпоинте `auth-service`. Реестр клиентов (`client_id`, хэш `client_secret`, разрешённые scopes) ведётся в auth-service.

## Последствия

- Единый источник всех JWT (user и service) — auth-service; downstream-сервисы используют один и тот же механизм валидации.
- При добавлении нового сервиса-клиента изменения локализованы в auth-service (реестр клиентов).
- Реестр клиентов в MVP заполняется через seed-данные; admin-эндпоинты управления клиентами не предусмотрены.
