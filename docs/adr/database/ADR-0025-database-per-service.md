# ADR-0025: Database-per-Service — изоляция данных по сервисам

- **Статус**: принят
- **Дата**: 2026-06-08

## Контекст

При общей БД между сервисами возникает неявная связанность через схему данных, что затрудняет независимое развёртывание и изменение сервисов. Альтернативы: shared database (одна БД для всех) — отклонено из-за связанности; shared schema — отклонено по той же причине.

## Решение

Каждый сервис владеет собственной базой данных и несёт полную ответственность за её состояние. Только сервис-владелец имеет доступ к своей БД; другие сервисы получают данные исключительно через API сервиса-владельца.

Исключение: сервисы-миграторы данных, заточенные под конкретную БД и знающие её схему, могут получать прямой доступ к данным помимо сервиса-владельца.

Именование БД: `<service>` в lowercase snake_case (например, `recipes_api`, `auth_service`).

Внутри БД данные разделяются по схемам: один bounded context — одна схема. Дефолтная схема (`public` в PostgreSQL) не используется. Именование схемы соответствует bounded context в lowercase snake_case.

## Последствия

- Сервисы независимы на уровне данных — изменение схемы одного сервиса не затрагивает другие.
- Данные доступны только через API владельца — обеспечивается инкапсуляция.
- Увеличивается количество БД в Docker Compose (по одной на сервис).
- Межсервисные запросы данных идут через HTTP API, а не через JOIN.
