# ADR-0014: EF Core как ORM

- **Статус**: принят
- **Дата**: 2026-06-07

## Контекст

Backend-сервисы реализуются на .NET 10 / C# (ADR-0011) и используют PostgreSQL (ADR-0004). Необходим единый способ доступа к данным с поддержкой миграций, маппинга на доменную модель (DDD, ADR-0012) и интеграции с гексагональной архитектурой (ADR-0013). Альтернативы: Dapper (микро-ORM, не предоставляет миграций и маппинга на агрегаты «из коробки»), чистый Npgsql/ADO.NET (максимальный контроль, но высокая стоимость поддержки) — отклонены.

## Решение

Основной ORM — **EF Core** (версия, поставляемая с .NET 10) с провайдером **Npgsql**.

- EF Core используется только внутри outbound-адаптера persistence (`Adapters/Postgresql`).
- DbContext, конфигурации сущностей, миграции — внутри адаптера.
- Доменная модель не знает про EF Core; маппинг — через configuration-классы (`IEntityTypeConfiguration<T>`).

## Последствия

- Миграции схемы БД выполняются средствами EF Core.
- Доступен LINQ и change tracking, но необходимо контролировать N+1 и leaky abstractions из домена в инфраструктуру.
- При сценариях с критичной производительностью допустимо точечное использование Dapper/raw SQL внутри того же адаптера.
