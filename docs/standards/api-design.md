# Стандарт: REST API Design

Источник: ADR-0006, ADR-0023, ADR-0024

## Правила

### Contract-First

- API проектируется по принципу contract-first: сначала создаётся OpenAPI-спецификация, только после — реализация.
- Файлы спецификаций: `docs/contracts/<bounded-context>/<service-name>.yaml`.
- Формат файлов — YAML, версия OpenAPI ≥ 3.0.0.
- Спецификация не должна содержать платформо-специфичных артефактов (.NET-дискриминаторы `$type`, полные имена типов и т.п.).

### Кодировка и формат

- Кодировка: UTF-8.
- Основной тип контента: `application/json`.
- JSON-поля именуются в camelCase.
- Необязательные поля без значений в JSON не включаются.

### Формат URI

```
{VERB} /api/{resource}/v{major_version}/{id}/{sub_resource}?{query_parameters}
```

- `{resource}` — существительное во множественном числе, kebab-case (например, `recipes`, `shopping-lists`).
- `{major_version}` — мажорная версия API (например, `v1`).
- `{id}` — идентификатор конкретного экземпляра ресурса (необязательный).
- `{sub_resource}` — вложенный/дочерний ресурс (необязательный); используется только для композиционно связанных ресурсов.
- `{query_parameters}` — параметры запроса (необязательные).

### Маршрутизация через API Gateway

Маршруты на gateway строятся по шаблону:

```
{gateway}/api/{bounded-context}/... → {service}/api/...
```

Gateway стрипает префикс `/{bounded-context}` перед проксированием. Примеры:

- `/api/recipes/**` → `recipes-service/api/...`
- `/api/auth/**` → `auth-service/api/...`

### HTTP-методы

- `GET` — получение данных ресурса.
- `POST` — создание ресурса; в редких случаях — получение данных при невозможности передать параметры через query.
- `PUT` — обновление ресурса (идемпотентно).
- `DELETE` — удаление ресурса.

Ограничения:

- `POST` для получения данных допустим только при большом объёме параметров запроса.
- Выбор между `POST` и `PUT` основывается на принципе идемпотентности: `POST` всегда изменяет состояние, `PUT` — только при первом применении.

### Параметры запроса

- Имена параметров — camelCase.
- Массив передаётся отдельными элементами через `&`: `ids={id1}&ids={id2}`.

### Статусные коды

| Код | Описание |
|-----|----------|
| 200 | OK — запрос выполнен, ответ содержит данные. |
| 201 | Created — ресурс создан (POST); ответ содержит идентификатор. |
| 202 | Accepted — запрос принят на асинхронную обработку. |
| 204 | No Content — запрос выполнен без тела ответа (UPDATE, DELETE). |
| 400 | Bad Request — ошибки в запросе или нарушение логики. |
| 401 | Unauthorized — пользователь не авторизован. |
| 403 | Forbidden — нет прав для выполнения запроса. |
| 404 | Not Found — некорректный URL или несуществующий маршрут (возвращается HTTP-сервером, не из кода). |
| 500 | Server Error — ошибка на стороне сервера. |
| 502 | Bad Gateway — недействительный ответ от вышестоящего сервера (применимо для API Gateway). |
| 503 | Service Unavailable — сервис временно недоступен. |

Примечания:

- `404` из программного кода не возвращается; если path-параметр некорректен — `400`.
- Если GET-запрос не возвращает данных — `200 OK` с пустым массивом/объектом.

### Коды ошибок в цепочке вызовов

| Ответ downstream | Ответ upstream |
|-----------------|----------------|
| 400 | 500 (ошибка бизнес-логики upstream) |
| 401 | 401 |
| 403 | 403 |
| 404 | 500 (ошибка конфигурации или бизнес-логики) |
| 500 | 500 |
| 503 | 503 |
| Недоступен | 503 |

### Формат ошибок

Все ошибочные ответы (4xx и 5xx) сериализуются по RFC 7807 с `Content-Type: application/problem+json` и обязательными полями `type`, `title`, `status`, `detail`, `instance`. Правило распространяется на все backend-сервисы, BFF и API Gateway.

### Типы данных

- Дата и время: UTC, формат ISO 8601.
- Временная зона (при необходимости): query-параметр `utcOffset`, формат `{+|-}hh:mm`.
- GUID: строка в инвариантном формате, например `aad1f839-df31-49f2-84d7-4dd01e04be77`.
- Enum: строковые значения в нижнем регистре.

### Локализация

- Локализация определяется по заголовку `Accept-Language`.
- `ru-RU` → русская локализация; не определена → английская.

### Аутентификация

- Защищённые эндпоинты требуют заголовка `Authorization: Bearer <JWT>`.
- Публичные эндпоинты доступны без токена.
