# Branch Plan: релизная политика и модель ветвления

## Роль

- AI-агент (Cline), создающий архитектурную документацию для монорепозитория.

## Контекст

- Проект `otus-06-cookbook` — учебный монорепозиторий «Книга рецептов».
- Используется монорепозиторий (ADR-0002: [docs/adr/general/ADR-0002-monorepo.md](../../adr/general/ADR-0002-monorepo.md)).
- Релизная политика: непрерывная доставка (Continuous Delivery) — каждый коммит в `main` потенциально готов к деплою.
- Модель ветвления: GitHub Flow — простая, подходит для учебного проекта с непрерывной доставкой.

## Задача

1. Использовать скил `adr` — создать `ADR-0003-release-and-branching.md` в `docs/adr/general/`:
   - Домен: `general`.
   - Контекст: нужна единая политика выпуска изменений и модель ветвления для монорепозитория.
   - Рассмотренные варианты релизной политики: непрерывная доставка (Continuous Delivery), релизные ветки (Release Branches).
   - Рассмотренные варианты модели ветвления: GitHub Flow, Git Flow, Trunk-Based Development.
   - Решение: непрерывная доставка + GitHub Flow.
   - Обоснование: проект учебный, команда небольшая, сложные модели ветвления избыточны.
   - Ссылка на ADR-0002: [docs/adr/general/ADR-0002-monorepo.md](../../adr/general/ADR-0002-monorepo.md).
   - Стандарт реализации: `docs/standards/branch-plan.md`.

2. Использовать скил `update-architecture-from-standard` — создать `docs/standards/branch-plan.md`:
   - Описать правила работы с ветками по GitHub Flow.
   - Структура: `main` — защищённая ветка, все изменения через feature-ветки и Pull Request.
   - Соглашения об именовании веток: `<type>/<short-description>`, где `<type>`:
     - `feature` — новая функциональность
     - `bug` — исправление бага
     - `hotfix` — быстрый фикс в продакшн
     - `refactor` — рефакторинг
     - `chore` — технические изменения (зависимости, конфигурация и т.п.)
   - Примеры: `feature/add-recipe-api`, `bug/fix-auth-error`, `hotfix/critical-null-ref`.
   - Правила PR: обязательный review, линейная история (squash или rebase).
   - Добавить ссылку на ADR-0003.

3. Обновить `architecture.md`:
   - Добавить ADR-0003 в индекс ADR.
   - Добавить `branch-plan.md` в индекс стандартов.

## Требования

- ADR фиксирует только решение — не описывает детали правил ветвления.
- Standard описывает конкретные правила и ссылается на ADR-0003.
- Все документы на русском языке.
- Применять принципы экономии токенов.
