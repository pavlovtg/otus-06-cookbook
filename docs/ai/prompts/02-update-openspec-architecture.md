# Update OpenSpec Architecture

## Роль

- AI-агент, создающий систему архитектурной документации для репозитория с OpenSpec.

## Контекст

- Репозиторий использует OpenSpec (`openspec/`) со схемой `spec-driven-with-adr`.
- Существующие ADR лежат в `/adr/` (корень) — их нужно перенести в `docs/adr/`.
- Папки `docs/architecture/rules/` и `docs/standards/` существуют, но пусты или частично заполнены.
- Скилы агента хранятся в `.clinerules/skills/`.
- Стандарт оформления Markdown: `docs/standards/markdown-code-style.md`.
- Принципы экономии токенов: `.clinerules/language.md`.

## Задача

Создать полную систему архитектурной документации.

### 1. Структура документации

```
ARCHITECTURE.md                         # единая точка входа (корень репозитория)
docs/
├── adr/
│   ├── general/                        # общие ADR
│   │   └── ADR-0001-<kebab-name>.md
│   ├── <domain>/                       # домен: сервис, bounded context, технология
│   │   └── ADR-NNNN-<kebab-name>.md
│   └── archive/                        # устаревшие и заменённые ADR
├── architecture/
│   ├── overview.md                     # высокоуровневое описание
│   ├── components.md                   # модули, зоны ответственности, зависимости
│   ├── diagrams/                       # диаграммы архитектуры
│   └── rules/
│       ├── general/                    # общие AR
│       │   └── AR-0001-<kebab-name>.md
│       └── <domain>/                   # домен: сервис, bounded context, технология
│           └── AR-NNNN-<kebab-name>.md
└── standards/
    └── markdown-code-style.md          # существующий стандарт
```

### 2. Система доменов

Каждый ADR и AR принадлежит домену. Домен определяет подпапку.

**Правила определения домена:**

- Конкретный сервис → домен = имя сервиса (например, `order-service`).
- Несколько сервисов → группирующий bounded context, application или `backend`/`frontend`.
- Технология → название технологии (например, `dotnet`, `postgresql`).
- Общие решения → домен `general`.
- При сомнении — спросить у пользователя.

**Архивирование ADR:**

- Если ADR устаревает или заменяется новым — перенести файл из домена в `docs/adr/archive/`.
- После переноса обновить индекс в `ARCHITECTURE.md`.

**Удаление AR:**

- Если AR больше не актуально — удалить файл.
- После удаления обновить индекс в `ARCHITECTURE.md`.

### 3. Файл `ARCHITECTURE.md` (корень репозитория)

Единая точка входа для людей и AI-агентов. НЕ содержит полное описание архитектуры.

Содержит:

- Краткий обзор системы (2-3 предложения).
- Порядок приоритета документов при противоречиях:
  1. Явный запрос пользователя
  2. OpenSpec (`openspec/`)
  3. Архитектурные правила (`docs/architecture/rules/`)
  4. Стандарты (`docs/standards/`)
  5. ADR (`docs/adr/`)
  6. Существующий код
  7. Предположения агента
- Политика загрузки контекста для AI-агентов.
- Индекс архитектурных правил (AR) по доменам.
- Индекс стандартов (включая `docs/standards/markdown-code-style.md`).
- Индекс ADR по доменам.

**Политика загрузки контекста:**

Агент НЕ читает всю документацию. Алгоритм:

1. Прочитать `ARCHITECTURE.md`.
2. Определить тип задачи.
3. Загрузить только нужные AR и стандарты.
4. Читать ADR только если: меняется архитектура / есть ссылка из AR или стандарта / нужно понять обоснование.

### 4. Шаблоны

Шаблоны хранятся в `assets/` соответствующих скилов.

**Шаблон ADR** (`adr/assets/adr-template.md`):

```markdown
# ADR-NNNN: <Название>

- **Статус**: принят | устарел | заменён ADR-NNNN
- **Домен**: <домен>
- **Дата**: YYYY-MM-DD
- **Заменяет**: ADR-NNNN (если применимо)

## Контекст

## Рассмотренные варианты

## Решение

## Последствия
```

**Шаблон AR** (`ar/assets/ar-template.md`):

```markdown
# AR-NNNN: <Название>

- **Домен**: <домен>

## Назначение

## Когда применяется

## Правило

## Разрешено

## Запрещено

## Связанные ADR
```

**Шаблон стандарта** (`update-architecture-from-standard/assets/standard-template.md`):

```markdown
# <Название стандарта>

## Назначение и область применения

## Правила

## Связанные AR и ADR
```

**Шаблон диаграммы** (`diagram/assets/diagram-template.md`):

```markdown
# <Название диаграммы>

## Тип

## Описание

## Диаграмма

```plantuml
@startuml

@enduml
```

## Связанные компоненты

```

### 5. Скилы в `.clinerules/skills/`

Каждый скил обязан:

- Применять принципы экономии токенов из `.clinerules/language.md` во всех создаваемых документах.
- Создавать документы на русском языке.

**`adr`** — создание и архивирование ADR:

1. Определить домен ADR (правила — см. раздел «Система доменов»).
2. Создать файл в `docs/adr/<domain>/ADR-NNNN-<kebab-name>.md` по шаблону из `assets/adr-template.md`.
3. Если ADR устаревает или заменяется — перенести файл в `docs/adr/archive/`.
4. Обновить индекс ADR в `ARCHITECTURE.md`.

**`ar`** — создание и удаление AR:

1. Определить домен AR (правила — см. раздел «Система доменов»).
2. Создать файл в `docs/architecture/rules/<domain>/AR-NNNN-<kebab-name>.md` по шаблону из `assets/ar-template.md`.
3. Если AR больше не актуально — удалить файл.
4. Обновить индекс AR в `ARCHITECTURE.md`.

**`diagram`** — создание и обновление диаграмм:

- Все диаграммы создаются на **PlantUML**.
- Для архитектурных диаграмм использовать **C4 Model** (C4-PlantUML).
- Диаграммы хранятся в `docs/architecture/diagrams/<kebab-name>.md`.
- Ссылки на диаграмму добавляются в подходящий документ: `ARCHITECTURE.md`, `overview.md`, `components.md`, конкретный AR/ADR/стандарт — в зависимости от контекста. При сомнении — спросить у пользователя.

**`update-architecture-from-adr`** — при добавлении, устаревании или замене ADR:

1. Использовать скил `adr` для создания или архивирования ADR.
2. Проанализировать решение ADR и определить:
   - Нужно ли создать новое AR (если ADR вводит архитектурное ограничение или правило).
   - Нужно ли обновить существующие AR (если ADR меняет или отменяет ограничение).
   - Нужно ли создать новый стандарт (если ADR вводит требование к реализации).
   - Нужно ли обновить существующие стандарты (если ADR меняет требования к реализации).
3. Если необходимость создания AR или стандарта неочевидна — спросить у пользователя.
4. Если нужны изменения AR — использовать скил `update-architecture-from-ar`.
5. Если нужны изменения стандартов — использовать скил `update-architecture-from-standard`.
6. Обновить `docs/architecture/overview.md` и `components.md` если ADR влияет на структуру системы.

**`update-architecture-from-ar`** — при создании, изменении или удалении AR:

1. Использовать скил `ar` для создания или удаления AR.
2. Обновить ссылки в связанных стандартах (добавить или убрать ссылку на AR).
3. Обновить ссылки в связанных ADR (только ссылки, не содержимое ADR).
4. При необходимости создать или обновить диаграмму — использовать скил `diagram`.

**`update-architecture-from-standard`** — при создании или изменении стандарта:

1. Создать или обновить файл в `docs/standards/` по шаблону из `assets/standard-template.md`.
2. Обновить индекс стандартов в `ARCHITECTURE.md`.
3. Обновить ссылки в связанных AR.
4. Проверить диаграммы, связанные со стандартом, и обновить при необходимости — использовать скил `diagram`.

### 6. Перевод шаблонов OpenSpec на русский язык

Обновить шаблоны в `openspec/schemas/spec-driven-with-adr/templates/`.

**Важно:** заголовки и ключевые слова, используемые утилитой OpenSpec, переводить нельзя:

- `## ADDED Requirements`, `## MODIFIED Requirements`, `## REMOVED Requirements`, `## RENAMED Requirements`
- `### Requirement:`, `#### Scenario:`
- `**WHEN**`, `**THEN**`
- `**Reason**`, `**Migration**`

Переводить только: заголовки разделов в `proposal.md`, `design.md`, `adr.md`, `tasks.md` и комментарии-подсказки.

Привести все шаблоны в соответствие с `docs/standards/markdown-code-style.md`.

**`proposal.md`:**

```markdown
# Зачем

<!-- Мотивация изменения. Какую проблему решает? Почему сейчас? -->

## Что меняется

<!-- Конкретные изменения: новые возможности, модификации, удаления. -->

## Возможности

### Новые возможности

<!-- Вводимые возможности. Заменить <name> на kebab-case идентификатор. Каждая создаёт specs/<name>/spec.md -->

- `<name>`: <краткое описание>

### Изменяемые возможности

<!-- Существующие возможности с изменением ТРЕБОВАНИЙ (не реализации).
     Только если меняется поведение на уровне спецификации. Использовать имена из openspec/specs/. -->

- `<existing-name>`: <что меняется>

## Влияние

<!-- Затронутый код, API, зависимости, системы. -->
```

**`design.md`:**

```markdown
# <!-- change-name -->

## Контекст

<!-- Предпосылки и текущее состояние. -->

## Цели / Не-цели

**Цели:**

<!-- Что достигает этот дизайн. -->

**Не-цели:**

<!-- Что явно выходит за рамки. -->

## Решения

<!-- Ключевые технические решения и обоснование. -->

## Риски / Компромиссы

<!-- Известные риски и компромиссы. -->
```

**`adr.md`:**

```markdown
# NNNN. <Название решения>

- **Статус**: принят | устарел | заменён ADR-NNNN
- **Дата**: YYYY-MM-DD

<!-- Заменяет: ADR-XXXX (указать только если этот ADR заменяет предыдущий; иначе убрать строку) -->

## Контекст

<!-- Ограничения, предпосылки, причины принятия решения. Если ADR заменяет предыдущий — объяснить почему. -->

## Решение

<!-- Принятый выбор, сформулированный чётко и однозначно. -->

## Последствия

<!-- Положительные, отрицательные и нейтральные последствия. -->
```

**`tasks.md`:**

```markdown
# <!-- change-name -->

## 1. <!-- Название группы задач -->

- [ ] 1.1 <!-- Описание задачи -->
- [ ] 1.2 <!-- Описание задачи -->

## 2. <!-- Название группы задач -->

- [ ] 2.1 <!-- Описание задачи -->
- [ ] 2.2 <!-- Описание задачи -->

**`spec.md`** — не переводить (все заголовки используются утилитой).

### 7. Обновление OpenSpec

Обновить скилы OpenSpec в `.clinerules/skills/`, добавив инструкцию:

- Перед началом работы прочитать `ARCHITECTURE.md`.
- Следовать политике загрузки контекста.
- При создании `design.md` и ADR — проверять соответствие действующим AR.
- **Перед созданием `tasks.md`** — обновить архитектурную документацию:
  - Если `design.md` или ADR вводят новые архитектурные решения — использовать скил `update-architecture-from-adr`.
  - Если вводятся новые архитектурные ограничения — использовать скил `update-architecture-from-ar`.
  - Если вводятся новые требования к реализации — использовать скил `update-architecture-from-standard`.
  - Если нужны новые диаграммы — использовать скил `diagram`.
  - При сомнении в необходимости обновления — спросить у пользователя.

### 8. Перенос существующих ADR

Если в `/adr/` (корень) есть файлы:

- Определить домен каждого ADR.
- Перенести в `docs/adr/<domain>/`.
- Обновить индекс в `ARCHITECTURE.md`.

### 9. Первый ADR

Создать ADR об использовании документации в формате Markdown:

- Домен: `general`.
- Контекст: документация ведётся в Markdown для совместного использования людьми и AI-агентами.
- Решение: использовать Markdown как единый формат документации.
- Стандарт реализации: `docs/standards/markdown-code-style.md`.
- Добавить в стандарт ссылку на ADR.

### 10. Файл `.clinerules/architecture.md`

Создать файл с содержимым:

```markdown
# Архитектура

## Правила

- **В начале каждой задачи** — прочитать `ARCHITECTURE.md` и следовать политике загрузки контекста.
```

Это правило читается Cline автоматически при каждом старте задачи.

## Требования

- `ARCHITECTURE.md` — только навигация, без полного описания архитектуры.
- ADR иммутабельны после принятия; для изменения решения — новый ADR с полем «Заменяет».
- Устаревшие ADR переносятся в `archive/`, не удаляются.
- Устаревшие AR удаляются.
- AR — источник истины для архитектурных ограничений.
- Стандарты — источник истины для деталей реализации.
- OpenSpec — источник истины для требований и поведения системы.
- ADR объясняют причины, НЕ являются стандартами реализации.
- Каждый AR — небольшой, независимый, легко читаемый документ.
- Имена файлов: `AR-NNNN-<kebab-name>.md`, `ADR-NNNN-<kebab-name>.md`.
- Нумерация монотонная в рамках всего репозитория, не повторяется.
- Все документы на русском языке.
- Применять принципы экономии токенов из `.clinerules/language.md` во всех создаваемых документах и скилах.
