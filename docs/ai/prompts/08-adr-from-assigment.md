# Промт: ADR, стандарты и AR из задания

## Роль

- Архитектор, анализирующий задание и фиксирующий явные архитектурные решения.

## Контекст

- Входной документ: `docs/prd/PRD-0001-MVP.md`.
- Существующие ADR: `docs/adr/general/` (ADR-0001 — Markdown, ADR-0002 — монорепо, ADR-0003 — ветвление).
- Шаблон ADR: `openspec/schemas/spec-driven-with-adr/templates/adr.md`.
- Стандарты: `docs/standards/`.
- Архитектурные правила: `docs/architecture/rules/`.
- Обзор архитектуры: `docs/architecture/overview.md`, `docs/architecture/components.md`.

## Задача

Прочитать `docs/prd/PRD-0001-MVP.md` и за один запуск создать:

1. **ADR** для каждого явного архитектурного решения из задания.
2. **Стандарты** (`docs/standards/`) для требований, задающих конкретные правила реализации.
3. **AR** (`docs/architecture/rules/`) для ограничений, которые должны соблюдаться в коде и инфраструктуре.
4. Обновить `docs/architecture/overview.md` и `docs/architecture/components.md` — добавить описание системы на основе PRD.

### ADR для создания

Создать следующие ADR (домен `general`, нумерация от следующего свободного индекса):

- **PostgreSQL как СУБД** — выбор PostgreSQL вместо SQLite; обоснование: требование production-ready БД, поддержка конкурентных запросов.
- **JWT-аутентификация** — выбор JWT для авторизации; обоснование: stateless, подходит для REST API.
- **REST API** — выбор REST как протокола взаимодействия frontend–backend; обоснование: требование OpenAPI/Swagger документации.
- **Docker Compose как среда развёртывания** — выбор Docker Compose; обоснование: требование `docker compose up` без дополнительной настройки.

Формат каждого ADR:

```markdown
# ADR-XXXX: <Название>

- **Статус**: принят
- **Домен**: general
- **Дата**: <YYYY-MM-DD>

## Контекст

<Почему нужно было принять решение>

## Рассмотренные варианты

- **<Вариант 1>** — <краткое описание>.
- **<Вариант 2>** — <краткое описание>.

## Решение

<Что выбрано и почему>

## Последствия

- <Следствие 1>.
- <Следствие 2>.
```

### Стандарты для создания

Создать файлы в `docs/standards/`:

- **`api-design.md`** — REST API документируется через OpenAPI/Swagger; Swagger UI доступен по живому URL `/api-docs`; файл спецификации не является артефактом сдачи.
- **`docker-compose-standard.md`** — правила именования стека (`COMPOSE_PROJECT_NAME`), сервисов и контейнеров; требование запуска одной командой `docker compose up` без дополнительной настройки; порядок зависимостей сервисов (`depends_on`).
- **`testing.md`** — минимальное покрытие тестами ≥ 80%; не менее 10 unit/integration тестов; CI запускает тесты автоматически.
- **`ci-standard.md`** — CI pipeline на GitHub Actions; обязательные шаги: lint + тесты; сборка Docker-образа — опционально.

### AR для создания

Создать файлы в `docs/architecture/rules/general/`:

- **`AR-docker-compose-self-contained.md`** — приложение запускается командой `docker compose up` в чистом окружении без дополнительной настройки (переменных окружения, ручных миграций, внешних зависимостей).

Формат AR:

```markdown
# AR-XXXX: <Название>

- **Домен**: general
- **Статус**: активно

## Правило

<Формулировка правила — одно предложение>

## Обоснование

<Почему это правило существует>

## Нарушение

<Что считается нарушением>
```

### Обновление архитектурной документации

**`docs/architecture/overview.md`** — добавить:

- Назначение системы: «Книга рецептов» — full-stack приложение для хранения, поиска и планирования рецептов.
- Ключевые технологии: PostgreSQL, JWT, REST API, Docker Compose, GitHub Actions.
- Ссылки на новые ADR.

**`docs/architecture/components.md`** — добавить описание компонентов:

- **Backend** — REST API сервер, OpenAPI/Swagger документация.
- **Frontend** — веб-приложение, адаптивная вёрстка (desktop + mobile).
- **Database** — PostgreSQL, seed-данные при старте.
- **Infrastructure** — Docker Compose (backend + frontend + БД), GitHub Actions CI.

## Требования

- Читать `docs/standards/markdown-code-style.md` и соблюдать стандарт оформления Markdown.
- Нумерация ADR — от следующего свободного индекса в `docs/adr/general/`.
- Нумерация AR — от следующего свободного индекса в `docs/architecture/rules/general/`.
- Не дублировать уже существующие ADR (ADR-0001 — Markdown, ADR-0002 — монорепо, ADR-0003 — ветвление).
- Каждый артефакт создаётся как отдельный файл.
- Язык всех документов: русский.
