# План разработки MVP книги рецептов

## Роль

- Архитектор-планировщик, формирующий бэклог для кросс-функциональной команды.

## Контекст

- Источник требований: `docs/prd/PRD-0001-MVP.md`.
- Архитектурные решения, AR и стандарты уже зафиксированы: `ARCHITECTURE.md`, `docs/adr/`, `docs/architecture/`, `docs/standards/`.
- Бизнесовые этапы реализуются через OpenSpec (`openspec/`), технические — обычным промтингом.

## Задача

- Составить план разработки MVP в виде файла `docs/prd/development-plan.md`.
- Разбить работу на последовательность небольших, но законченных этапов (бизнесовых и технических).
- Для каждого этапа указать: номер, тип (бизнес/тех), название, краткое описание результата, зависимости и чекбокс выполнения.
- Учесть зависимости: фича не планируется раньше её предпосылок (теги требуют рецептов, планировщик меню — рецептов и пользователя, список покупок — планировщика, комментарии и рейтинг — рецептов и аутентификации и т.д.).

## Требования

- Язык: русский, принципы экономии токенов.
- Формат: Markdown, сквозная нумерация, чекбоксы `- [ ]` для отметки выполнения.
- Уровень детализации — бэклог кросс-функциональной команды: НЕ упоминать участки кода, классы, файлы тестов, конкретные эндпоинты.
- Размер этапа — один законченный блок.
  - Примеры бизнес-этапов: «CRUD рецептов», «поиск по рецептам», «категории и теги», «планировщик меню», «список покупок».
  - Примеры тех-этапов: «настроить CI pipeline», «подготовить seed-данные», «написать `REPORT.md`».
- Если для реализации бизнес-этапа требуется новое архитектурное решение (выбор технологии, протокола, подхода) — выделить его отдельным предшествующим тех-этапом «зафиксировать ADR …».
- Если для реализации бизнес-этапа требуется UX-проработка интерфейса — выделить отдельный предшествующий этап «сделать макет …» (wireframe / прототип).
- Бизнес-этапы покрывают все функциональные требования PRD; технические — нефункциональные (CI, Docker Compose, OpenAPI, seed-данные, документация, новые ADR, макеты).
- Зависимости указывать ссылками на номера предыдущих этапов; если зависимостей нет — «нет».
- Структура каждого этапа:
  - заголовок `## N. [бизнес|тех] Название`;
  - строка статуса `- [ ] не выполнено`;
  - `Описание:` 1–3 строки;
  - `Зависимости:` перечень номеров или «нет».
- В начале файла — короткая преамбула (1 абзац) и легенда типов этапов.
