# Упрощение skills архитектурной документации

## Роль

- Архитектор-рефакторер, упрощающий систему skills для управления архитектурными артефактами.

## Контекст

- Текущие skills: `adr`, `ar`, `update-architecture-from-adr`, `update-architecture-from-ar`, `update-architecture-from-standard`, `diagram`.
- Триггеры в `.clinerules/architecture.md` ссылаются на `update-architecture-from-*`.
- Артефакты архитектуры: ADR (`docs/adr/`), AR (`docs/architecture/rules/`), стандарты (`docs/standards/`), диаграммы (`docs/architecture/diagrams/`).
- Индекс — `ARCHITECTURE.md`. Обзор — `docs/architecture/overview.md`, `docs/architecture/components.md`.

### Проблемы текущей структуры

- Искусственное разделение пар `adr` / `update-architecture-from-adr` (и аналогично для `ar`) — граница между skills размыта, второй просто оркеструет первый.
- Асимметрия: для стандарта нет «низкого» skill, только `update-architecture-from-standard`.
- Триггеры в `.clinerules/architecture.md` дублируют логику skills.
- `diagram` не включён в триггеры.

## Задача

Привести систему skills к симметричной структуре «один артефакт — один skill». Каждый skill — самодостаточен.

### Целевая структура

**4 skills:**

| Skill | Артефакт | Операции |
|-------|----------|----------|
| `adr` | ADR | создание, архивирование |
| `ar` | AR | создание, удаление |
| `standard` | стандарт | создание, обновление, удаление |
| `diagram` | диаграмма | создание, обновление |

Каждый skill сам отвечает за **консистентность связанных артефактов** (индекс в `ARCHITECTURE.md`, перекрёстные ссылки, `overview.md`, `components.md`).

### Принцип размещения правил

- Общие правила (`Домен`, `Нумерация`, `Консистентность`) **живут внутри каждого skill**, а не в `.clinerules/architecture.md`.
- Причина: правила загружаются только при активации skill — токены базового контекста не тратятся.
- Дублирование между skills допустимо и осознанно.

### Шаги

1. **Обновить `.clinerules/architecture.md`** — только триггеры, без общих принципов:

   ```markdown
   # Архитектура

   ## Правила

   - В начале каждой задачи — прочитать `ARCHITECTURE.md` и следовать политике загрузки контекста.
   - При создании/редактировании Markdown — соблюдать `docs/standards/markdown-code-style.md`.
   - При любой операции с ADR — использовать skill `adr`.
   - При любой операции с AR — использовать skill `ar`.
   - При любой операции со стандартом — использовать skill `standard`.
   - При любой операции с диаграммой — использовать skill `diagram`.
   ```

2. **Переписать `.clinerules/skills/adr/SKILL.md`** — объединить логику текущих `adr` и `update-architecture-from-adr`. Skill самодостаточен: содержит блоки «Определение домена», «Нумерация», «Создание ADR», «Архивирование ADR», «Консистентность», «Иммутабельность».

3. **Переписать `.clinerules/skills/ar/SKILL.md`** — объединить с `update-architecture-from-ar`. Блоки: «Определение домена», «Нумерация», «Создание AR», «Удаление AR» (вместо архивирования), «Консистентность», «Принципы» (маленькие независимые правила).

4. **Создать `.clinerules/skills/standard/SKILL.md`** — переработать `update-architecture-from-standard`. Блоки: «Путь» (без нумерации), «Создание/Обновление/Удаление стандарта», «Консистентность», «Принципы» (различие между ADR/AR/стандартом).

5. **Обновить `.clinerules/skills/diagram/SKILL.md`** — добавить явный блок «Консистентность» (куда добавлять ссылку: `ARCHITECTURE.md`, `overview.md`, `components.md` или конкретный ADR/AR/стандарт).

6. **Удалить устаревшие skills:**

   - `.clinerules/skills/update-architecture-from-adr/`
   - `.clinerules/skills/update-architecture-from-ar/`
   - `.clinerules/skills/update-architecture-from-standard/` (assets перенести в `standard/assets/`).

7. **Перенести шаблоны:**

   - `update-architecture-from-standard/assets/standard-template.md` → `standard/assets/standard-template.md`.
   - Остальные `assets/` остаются в своих skills.

8. **Актуализировать индексы в `ARCHITECTURE.md`** — таблицы могут быть неактуальны после предыдущих промтов. Проверить и заполнить:
   - Таблица AR — все файлы из `docs/architecture/rules/`.
   - Таблица стандартов — все файлы из `docs/standards/`.
   - Таблица ADR — все файлы из `docs/adr/` (включая `archive/` со статусом).

9. **Обновить memory bank** (`activeContext.md`, `progress.md`) — зафиксировать рефакторинг.

### Шаблон каждого skill

```markdown
---
name: <имя>
description: <одна фраза о том, когда вызывать>
---

Перед началом — прочитать `ARCHITECTURE.md`.

## Определение домена  <!-- только для adr/ar -->

- Конкретный сервис → имя сервиса.
- Несколько сервисов → bounded context, `backend` или `frontend`.
- Технология → имя технологии.
- Общее → `general`.
- При сомнении — спросить у пользователя.

## Нумерация  <!-- только для adr/ar -->

- Монотонная в рамках всего репозитория.
- Следующий номер = max существующий + 1 (включая `archive/`).

## Создание <артефакт>

1. Определить домен и номер (если применимо).
2. Создать файл по шаблону `assets/<артефакт>-template.md`.
3. Применить шаг «Консистентность».

## <Спец-операция: архивирование / удаление / обновление>

<Описание специфики артефакта>

## Консистентность

При любом изменении:

1. Обновить индекс соответствующего раздела в `ARCHITECTURE.md`.
2. Обновить перекрёстные ссылки в связанных артефактах.
3. Обновить `docs/architecture/overview.md`, если влияет на структуру.
4. Обновить `docs/architecture/components.md`, если влияет на компоненты.

## Специфика / Принципы

<Иммутабельность для ADR / независимость для AR / различие ADR↔AR↔стандарт и т.п.>
```

## Требования

- Соблюдать `docs/standards/markdown-code-style.md`.
- Общие правила (`Домен`, `Нумерация`, `Консистентность`) — внутри каждого skill, не в `architecture.md`. Дублирование между skills допустимо.
- Сохранить ссылки на существующие шаблоны (`assets/*-template.md`).
- Не менять содержимое уже созданных артефактов (ADR-0001..0007, AR-0001..0002, стандарты).
- После рефакторинга проверить, что `ARCHITECTURE.md` содержит актуальные индексы ADR/AR/стандартов.
- Язык всех документов: русский.
- Принципы экономии токенов: одно предложение вместо абзаца, буллеты вместо прозы.
