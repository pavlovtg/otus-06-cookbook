## Исходные требования

### Описание проекта

Создать full-stack приложение (backend + frontend + БД), используя AI для генерации архитектуры, кода и инфраструктуры.
Проект стартует с описания функционала, а не с кода.

**Ожидаемые результаты:**

- Работающее приложение с минимум 4–5 ключевыми фичами.
- Тесты, документация и CI pipeline.
- Демо-развёртывание (Docker Compose, локально).

**Технологии:**

- Не ограничено, если можно запаковать в Docker Compose.
- PostgreSQL / SQLite для простоты.
- Docker, GitHub Actions.

### Общие функциональные требования

- CRUD основных сущностей с валидацией на клиенте и сервере.
- Поиск и фильтрация по ключевым полям.
- Дашборд с визуализацией данных (графики / статистика).
- Пагинация списков.
- Адаптивная вёрстка (desktop + mobile).

### Общие нефункциональные требования

- Seed-данные: приложение запускается с реалистичным набором демо-данных.
- REST API с OpenAPI/Swagger документацией (живая веб-страница, не файл в проекте).
- Не менее 10 unit/integration тестов.
- CI pipeline (GitHub Actions): lint + тесты; сборка Docker-образа — опционально.
- Всё приложение запускается одной командой `docker compose up` (backend + frontend + БД).
- `README.md` с инструкцией по запуску (формат — Markdown).
- `ARCHITECTURE.md` с описанием архитектуры и плана разработки (создаётся до начала кодирования, формат — Markdown).
- `REPORT.md` с историей создания проекта: ключевые проблемы, решения, удачные и неудачные шаги. Заполняется по мере работы, не в конце. Формат — Markdown.
- Проект в новом пустом Git-репозитории с адекватной историей коммитов.
- (Опционально) Ссылка на видео-демонстрацию.

### Вариант 3: Книга рецептов — функциональные требования

- Рецепты: название, описание, время приготовления, сложность, фото (upload), пошаговые инструкции.
- Ингредиенты с количеством и единицами измерения (граммы, штуки, мл).
- Категории и теги (завтрак, десерт, веган, быстрое и т.д.).
- Поиск рецептов по названию, ингредиентам («что приготовить из курицы и риса»), тегам.
- Планировщик меню на неделю: drag-and-drop рецептов в слоты (пн–вс × завтрак/обед/ужин).
- Автогенерация списка покупок из плана меню (агрегация ингредиентов с суммированием).
- Масштабирование порций: пересчёт ингредиентов на N порций.
- Избранное: добавление рецептов в закладки.
- Рейтинг рецептов (1–5 звёзд) с сортировкой по среднему рейтингу.
- Регистрация / авторизация (JWT), два предзаполненных пользователя; рецепт может быть публичным (виден всем) или приватным (только автор); имя автора отображается на карточке рецепта.
- Комментарии к рецептам: авторизованный пользователь оставляет комментарии с отметкой времени и автора; владелец рецепта может удалить любой комментарий.

### Seed-данные

- 25+ рецептов с фото-заглушками.
- 50+ ингредиентов.
- Готовый план меню на неделю.
- 2 пользователя.
- 20+ комментариев.

### Критерии оценки

- Выполнены все общие требования (CRUD, поиск, дашборд, пагинация, адаптивная вёрстка, тесты, CI, Docker Compose, OpenAPI, README).
- Выполнены все функциональные требования варианта «Книга рецептов».
- Приложение поставляется с seed-данными указанного объёма.
- Docker Compose — приложение запускается в чистом окружении одной командой `docker compose up` без дополнительной настройки (блокирующий критерий).
- AI-процесс разработки — `ARCHITECTURE.md` создан до начала кодирования (проверяется по git-истории); `REPORT.md` содержит описание использованных AI-инструментов и минимум один показательный пример промпта с кратким разбором.

### Процесс сдачи

1. Студент предоставляет ссылку на GitHub-репозиторий.
2. Преподаватель выполняет:

   ```bash
   git clone <repo> && cd <repo>
   docker compose up
   ```

3. Преподаватель открывает приложение в браузере и проходит по основным сценариям (seed-данные уже загружены).
4. Преподаватель открывает Swagger UI по адресу `http://localhost:<port>/docs` (или `/redoc`, `/api-docs`) и проверяет эндпоинты.
5. Преподаватель проверяет, что pipeline в GitHub Actions проходит (lint + тесты + сборка).
6. Демо (5 мин) — студент показывает приложение, объясняет архитектуру и ключевые решения.

## Задача

### 1. Создать `docs/prd/PRD-0001-MVP.md`

Собрать PRD из раздела «Исходные требования». Структура документа (в указанном порядке):

- **Цель проекта** — краткое описание и ожидаемые результаты.
- **Вариант: Книга рецептов** — специфика выбранного варианта.
- **Проблема** — формулировка боли целевой аудитории: домашние повара хранят рецепты в разрозненных источниках, не могут планировать питание и составлять списки покупок.
- **Целевая аудитория** — одна персона: домашний повар (готовит дома для себя и семьи, хочет хранить рецепты, планировать меню, составлять список покупок, искать рецепты по ингредиентам).
- **Функциональные требования** — общие + специфичные для варианта.
- **Нефункциональные требования** — технические ограничения и стандарты.
- **Out of Scope** — что явно не входит в MVP: мобильное приложение, социальная сеть, монетизация, интеграции с магазинами, импорт рецептов с внешних сайтов.
- **Seed-данные** — объём и состав демо-данных.
- **Критерии оценки** — условия приёмки.
- **Метрики успеха** — покрытие тестами ≥ 80%, все критерии сдачи выполнены, приложение запускается за < 2 минуты в чистом окружении.
- **Процесс сдачи** — шаги проверки преподавателем.
