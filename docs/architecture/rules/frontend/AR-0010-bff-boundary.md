# AR-0010: BFF boundary

Источник: ADR-0015, ADR-0017

## Правило

- BFF выполняет ТОЛЬКО cross-cutting и адаптационные задачи: проксирование REST-запросов в API Gateway, агрегацию ответов нескольких backend-эндпоинтов под форму UI, управление сессией, CSRF-защиту, заголовки безопасности, преобразование DTO backend → ViewModel UI, валидацию входа форм через Zod.
- BFF stateless: состояние сессии — только в **signed encrypted cookie** (httpOnly, Secure, SameSite=Lax минимум; подписан и зашифрован секретом, известным только BFF; содержит JWT backend или достаточные claims). Любой инстанс BFF обрабатывает запрос без обращения к внешнему хранилищу.
- JWT не покидает серверную сторону BFF. В браузер уходит только httpOnly session cookie. Браузер не отправляет `Authorization: Bearer ...` напрямую в API Gateway — все запросы идут через BFF.
- UI и BFF — один Next.js-процесс (контейнер `web`). Серверная логика BFF размещается в выделенных модулях (`lib/bff/`, `app/api/*/route.ts`, server actions); BFF-модули не импортируются из client-компонентов — нарушение ловится ESLint-правилом `import/no-restricted-paths` или эквивалентом.

## Запрещено

- Доменная/бизнес-логика в BFF (доменные инварианты, бизнес-вычисления, которые должны переживать смену клиента).
- Прямой доступ к БД из BFF (PostgreSQL доступна только backend-сервисам).
- Сессии в Redis, БД, иных внешних хранилищах; серверная in-memory сессия, привязанная к конкретному инстансу BFF.
- JWT в `localStorage`, `sessionStorage`, нешифрованном cookie, JSON-ответах BFF клиенту, JS-бандле, HTML; прокидывание raw JWT в client-side React-компоненты.
- Импорт BFF-модулей (содержащих секреты, токены, прямые fetch к gateway) в client-компоненты.
- Выделение BFF в отдельный процесс или сервис без нового ADR.
