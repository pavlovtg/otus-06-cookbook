# AR-0009: Видимость типов — `internal` по умолчанию

- **Домен**: backend
- **Статус**: активно

## Правило

В backend-сборках все типы по умолчанию объявляются с модификатором `internal`.

Модификатор `public` допустим **только** для типов, являющихся **публичным API сборки** — то есть типов, на которые ссылаются из других сборок.

- Сервисы (executable, `Program.cs`) и утилиты (CLI-инструменты) **не имеют публичного API** — все их типы должны быть `internal`.
- Библиотеки (`*.csproj` без точки входа, переиспользуемые между сборками) имеют публичное API только в виде типов, действительно используемых снаружи. Остальные типы — `internal`.
- Для доступа из тестовых сборок к `internal`-типам использовать `InternalsVisibleTo` в `AssemblyInfo.cs` или `<InternalsVisibleTo>` в `.csproj`.

## Обоснование

- Минимизация публичной поверхности упрощает рефакторинг и эволюцию: `internal`-типы можно менять без риска поломать внешних потребителей.
- В сервисах нет «внешних потребителей» на уровне типов — наружу торчат только HTTP/gRPC-эндпоинты и контракты сообщений, а не CLR-типы.
- Делает архитектурные намерения явными: `public` сигнализирует «это часть контракта сборки».

## Нарушение

- `public` у типа в сборке-сервисе или сборке-утилите.
- `public` у внутренней реализации в библиотеке, не использующейся из других сборок.
- Тестовый проект напрямую ссылается на `public`-проброс типов, сделанный «чтобы тесты увидели».
