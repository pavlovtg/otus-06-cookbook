# AR-0008: Доменная логика — через DDD-паттерны

- **Домен**: backend
- **Статус**: активно

## Правило

Доменная логика backend-сервисов выражается через тактические паттерны Domain-Driven Design:

- бизнес-правила и инварианты живут внутри агрегатов (Aggregate Root);
- состояние агрегата изменяется только через его методы — анемичная модель запрещена;
- идентификаторы агрегатов и сущностей — типизированные Value Objects, не «голые» примитивы;
- значимые понятия домена без идентичности моделируются как Value Objects (иммутабельные, с проверкой инвариантов в конструкторе);
- межагрегатные взаимодействия — через доменные события и application-сервисы, а не через прямые ссылки между агрегатами.

## Обоснование

DDD ([ADR-0012](../../../adr/backend/ADR-0012-domain-driven-design.md)) обеспечивает явное выражение бизнес-правил, защищает инварианты и масштабируется на сложный домен проекта.

## Нарушение

- Публичные сеттеры на свойствах агрегатов и сущностей.
- Бизнес-логика в контроллерах, application-сервисах, репозиториях вместо агрегатов.
- Использование `int` / `Guid` / `string` напрямую вместо доменных Value Objects для идентификаторов и значимых понятий.
- Прямые ссылки между разными агрегатами вместо обращения по идентификатору.
