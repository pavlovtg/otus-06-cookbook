# recipe-ingredients

## Контекст

Агрегат `Recipe` существует, но не содержит ингредиентов. Сущность `Ingredient` (каталог) уже реализована. Необходимо связать рецепт с ингредиентами через промежуточную структуру, хранящую количество. Изменение затрагивает backend (домен, БД, API) и frontend (схемы, BFF, UI).

Действующие архитектурные ограничения: DDD-агрегаты (AR-0007), гексагональная архитектура (AR-0006), EF Core конфигурации через `IEntityTypeConfiguration` (AR-0051), contract-first OpenAPI (AR-0015), Zod-валидация на фронте (ADR-0019).

## Цели / Не-цели

**Цели:**

- Добавить список ингредиентов с количеством к агрегату `Recipe`.
- Отобразить ингредиенты на детальной странице рецепта.
- Поддержать управление ингредиентами в форме создания/редактирования рецепта.
- Заблокировать удаление ингредиента из каталога, если он используется в рецептах.
- Ввести две формы DTO рецепта: краткую (список) и полную (CRUD/детальная).

**Не-цели:**

- Масштабирование порций (пересчёт количества) — отдельная фича.
- Поиск рецептов по ингредиентам — отдельная фича.
- Список покупок — отдельная фича.

## Решения

### 1. RecipeIngredient как owned entity агрегата Recipe

`RecipeIngredient` — Value Object внутри агрегата `Recipe`, содержащий `IngredientId` и `amount` (decimal). Хранится в отдельной таблице `cookbook.recipe_ingredients` как owned collection EF Core.

**Альтернатива**: отдельный агрегат `RecipeIngredient` — избыточно, у него нет собственного жизненного цикла вне рецепта.

### 2. Две формы DTO рецепта: RecipeShortDto и RecipeDto

- `RecipeShortDto` — краткая форма для `GET /api/v1/recipes` (список): `id`, `title`, `cookingTime`, `difficulty`, `servings`. Без ингредиентов — список рецептов не требует их отображения.
- `RecipeDto` — полная форма для `GET /api/v1/recipes/{id}`, `POST`, `PUT`: все поля включая `ingredients: RecipeIngredientDto[]`.

Это исключает избыточную загрузку ингредиентов при получении списка рецептов.

**Альтернатива**: одна схема с опциональным полем `ingredients` — неоднозначно, клиент не знает, когда ожидать поле.

### 3. Замена коллекции ингредиентов при обновлении рецепта

При `PUT /api/v1/recipes/{id}` коллекция ингредиентов полностью заменяется новым списком из запроса (delete-and-insert). Это упрощает логику обновления и соответствует семантике PUT.

**Альтернатива**: патч-операции (добавить/удалить отдельный ингредиент) — усложняет API и клиент без значимой выгоды на данном этапе.

### 4. Блокировка удаления ингредиента → 400 Bad Request

При удалении ингредиента из каталога `IngredientService` проверяет через репозиторий, используется ли ингредиент в рецептах. Если да — выбрасывает доменное исключение `IngredientInUseException`. Контроллер перехватывает его и возвращает `400 Bad Request` с Problem+JSON.

**Альтернатива**: `409 Conflict` — семантически точнее, но в проекте принято возвращать `400` для всех доменных ошибок (паттерн уже установлен в `RecipesController`).

### 5. Единица измерения берётся из ингредиента

`unit` не хранится в `recipe_ingredients` — берётся из связанного `Ingredient`. Это исключает рассинхронизацию единиц.

### 6. Расширение OpenAPI контракта

- Добавляется `RecipeShortDto` (без ингредиентов) — используется в `GET /api/v1/recipes`.
- `RecipeDto` расширяется полем `ingredients: RecipeIngredientDto[]`.
- `RecipeRequest` получает необязательное поле `ingredients: RecipeIngredientRequest[]` (по умолчанию пустой список).
- Добавляются схемы `RecipeIngredientDto` и `RecipeIngredientRequest`.

## Риски / Компромиссы

- [Миграция данных] Существующие рецепты получат пустой список ингредиентов → приемлемо, seed-данные обновляются.
- [Целостность] FK constraint в таблице `recipe_ingredients` → `ingredients` как дополнительный барьер целостности данных.
- [Breaking change] `GET /api/v1/recipes` теперь возвращает `RecipeShortDto` вместо `RecipeDto` — клиенты, ожидающие полную форму, сломаются. Митигация: обновить все клиенты (BFF) одновременно с backend.

## План миграции

1. Обновить OpenAPI контракт (`docs/contracts/cookbook/recipes.yaml`).
2. Добавить `RecipeIngredient` в домен и агрегат `Recipe`.
3. Добавить конфигурацию EF Core и создать миграцию.
4. Обновить репозиторий: `GetRecipesAsync` без Include ингредиентов, `GetByIdAsync` с Include.
5. Обновить сервисы и контроллеры (две формы DTO, проверка при удалении ингредиента).
6. Обновить seed-данные.
7. Обновить Zod-схемы, BFF и UI.
8. Написать тесты.

Откат: миграция обратима (drop table `recipe_ingredients`), восстановить прежние DTO.
