# user-auth

## Зачем

Приложение не имеет системы аутентификации: все эндпоинты публичны, нет понятия «текущий пользователь». Без авторизации невозможно реализовать персонализированные функции (избранное, планировщик, комментарии) и разграничение доступа к рецептам.

## Что меняется

- Регистрация нового пользователя по email и паролю.
- Вход по email + пароль, получение сессии.
- Выход из системы.
- Получение профиля текущего пользователя.
- Защита маршрутов создания, редактирования и удаления рецептов — только для авторизованных.
- Два предзаполненных пользователя в seed-данных: обычный пользователь и администратор.
- Отображение имени текущего пользователя в шапке приложения; кнопки «Войти» / «Зарегистрироваться» / «Выйти».
- Страницы входа и регистрации в UI.

## Возможности

### Новые возможности

- `auth-register`: регистрация нового пользователя (email, display_name, password).
- `auth-login`: вход по email + password, установка сессии.
- `auth-logout`: выход, завершение сессии.
- `auth-me`: получение профиля текущего авторизованного пользователя.
- `auth-protected-routes`: защита маршрутов создания/редактирования/удаления рецептов.

### Изменяемые возможности

- `recipe-create`: создание рецепта доступно только авторизованным пользователям.
- `recipe-delete`: удаление рецепта доступно только авторизованным пользователям.
- `recipe-edit`: редактирование рецепта доступно только авторизованным пользователям.

## Влияние

- Новые API-эндпоинты: `POST /api/v1/auth/register`, `POST /api/v1/auth/login`, `POST /api/v1/auth/logout`, `GET /api/v1/auth/me`.
- Существующие эндпоинты создания/редактирования/удаления рецептов становятся защищёнными.
- Новая таблица пользователей в БД.
- Новые страницы и компоненты в UI.
- Переменная окружения для JWT-секрета.
