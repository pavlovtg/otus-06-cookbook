# recipe-photos

## Контекст

Агрегат `Recipe` существует, но не содержит фотографий. Архитектурные решения по хранению фото уже приняты и зафиксированы в ADR-0034 и AR-0059–AR-0062: фото хранятся в PostgreSQL (bytea) в отдельной таблице `recipe_photos`, thumbnail генерируется через ImageSharp, раздача через публичный endpoint без JWT, seed-фото через `CookbookSeeder`.

Дизайн UI следует макету `docs/design/mockup/` (тёмная тема, дизайн-токены из `styles.css`). Компоненты разрабатываются через Storybook (`docs/design/storybook/`).

Действующие ограничения: DDD-агрегаты (AR-0007), гексагональная архитектура (AR-0006), EF Core конфигурации через `IEntityTypeConfiguration` (AR-0051), contract-first OpenAPI (AR-0015), Zod-валидация на фронте (ADR-0019), фото управляются отдельно от основных полей рецепта.

## Цели / Не-цели

**Цели:**

- Добавить поле `photo_id` (nullable FK) в таблицу `recipes`.
- Реализовать таблицу `recipe_photos` с оригиналом и thumbnail (bytea).
- Добавить endpoints загрузки, удаления и раздачи фото.
- Расширить DTO рецепта идентификатором фото (`photoId: Guid?`, required, nullable).
- Отобразить фото (или заглушку) на карточке и детальной странице.
- Добавить управление фото в форму создания/редактирования рецепта.
- Загрузить seed-фото через `CookbookSeeder`.

**Не-цели:**

- Несколько фото на рецепт — один рецепт, одно фото.
- Хранение фото в файловой системе или S3.
- Изменение размера/кадрирования пользователем.

## Решения

### 1. Фото управляется отдельно от основных полей рецепта

Загрузка и удаление фото — отдельные endpoints (`POST /api/v1/recipes/{id}/photo`, `DELETE /api/v1/recipes/{id}/photo`), не часть `PUT /api/v1/recipes/{id}`. Это позволяет изменять поля рецепта без риска случайно затронуть фото.

**Альтернатива**: передавать фото в теле `PUT` как multipart — усложняет форму редактирования и нарушает принцип единственной ответственности endpoint.

### 2. DTO возвращает `photoId`, URL строит фронтенд

`RecipeDto` и `RecipeShortDto` получают поле `photoId: Guid?` (required, nullable). Фронтенд самостоятельно формирует URL по схеме `/api/cookbook/photos/{photoId}` и `/api/cookbook/photos/{photoId}/thumbnail`, используя `photoId`. Это позволяет BFF сформировать корректную ссылку, зная идентификатор фото, и исключает сложность формирования абсолютных URL на бэкенде.

**Альтернатива**: возвращать только `hasPhoto: bool` — BFF не может сформировать URL без `photoId`, потребовался бы дополнительный запрос по `recipeId`.

### 3. RecipePhoto как отдельная сущность домена

`RecipePhoto` — отдельная сущность с `RecipePhotoId`, `RecipeId`, `OriginalData` (byte[]) и `ThumbnailData` (byte[]). Агрегат `Recipe` хранит nullable `RecipePhotoId`. Жизненный цикл фото привязан к рецепту через CASCADE DELETE.

**Альтернатива**: хранить bytea прямо в `Recipe` — нарушает принцип единственной ответственности, усложняет загрузку рецептов без фото.

### 4. Репозиторий загружает только нужное поле

`GET /api/v1/photos/{photoId}` загружает только колонку `original_data`, не трогая `thumbnail_data`. `GET /api/v1/photos/{photoId}/thumbnail` загружает только `thumbnail_data`. Это исключает избыточную передачу данных из БД.

**Альтернатива**: загружать всю строку — при фото 10 МБ + thumbnail 200 КБ каждый запрос тянет лишние данные.

### 5. Thumbnail генерируется при загрузке через ImageSharp

При загрузке сервер генерирует thumbnail 400×300 px (crop center) через SixLabors.ImageSharp. Оригинал и thumbnail сохраняются в одной строке `recipe_photos`.

**Альтернатива**: генерировать thumbnail при каждом запросе — избыточная нагрузка на CPU.

### 6. Сброс кэша nginx при изменении фото

После успешной загрузки или удаления фото BFF отправляет nginx `PURGE`-запрос на URL фото и thumbnail. BFF находится во внутренней сети и имеет прямой доступ к nginx, что не нарушает изоляцию сервисов. Бэкенд о nginx не знает.

**Альтернатива**: не сбрасывать кэш — пользователь видит старое фото до 24 часов после замены.

### 7. Seed-фото из apps/seed/photos/

Seed-фотографии хранятся в `apps/seed/photos/` и загружаются через `CookbookSeeder` в той же транзакции, что и seed-рецепты. Операция идемпотентна (AR-0062).

## Риски / Компромиссы

- [Размер БД] Хранение bytea в PostgreSQL увеличивает размер БД. Для учебного проекта с 25+ рецептами и фото ≤ 10 МБ — приемлемо.
- [Производительность] Загрузка списка рецептов не тянет bytea — `RecipeShortDto` содержит только `hasPhoto`, данные фото загружаются отдельным запросом браузера по URL.
- [Breaking change] `RecipeDto` и `RecipeShortDto` расширяются новым полем — обратно совместимо (добавление поля).

## План миграции

2. Обновить OpenAPI контракт (`docs/contracts/cookbook/recipes.yaml`): новые endpoints, `photoId` в DTO.
3. Добавить `RecipePhoto` в домен, `RecipePhotoId` в `Recipe`.
4. Добавить `RecipePhotoConfiguration` (EF Core) и создать миграцию.
5. Реализовать `IRecipePhotoRepository` с раздельной загрузкой полей и адаптер PostgreSQL.
6. Реализовать Application-сервис загрузки/удаления фото с ImageSharp и сбросом кэша nginx.
7. Добавить `PhotosController` (раздача) и расширить `RecipesController` (upload/delete).
8. Обновить seed-данные: добавить фото в `apps/seed/photos/`, расширить `CookbookSeeder`.
9. Обновить Zod-схемы, BFF и UI (карточка, детальная страница, форма).
10. Написать тесты.

Откат: миграция обратима (drop column `photo_id`, drop table `recipe_photos`), восстановить прежние DTO.

## Тесты

- **Unit (backend)**: `RecipePhoto.Create` — валидация MIME и размера; `RecipePhotoService` — логика замены фото.
- **Unit (frontend)**: Zod-схемы с `hasPhoto`; BFF-функции загрузки/удаления фото.
- **Integration (DB)**: репозиторий — сохранение фото, загрузка только `original_data`, загрузка только `thumbnail_data`, CASCADE DELETE при удалении рецепта.
- **Microservice**: `POST /api/v1/recipes/{id}/photo` → `GET /api/v1/photos/{photoId}` возвращает фото; `DELETE /api/v1/recipes/{id}/photo` → `hasPhoto = false`; загрузка недопустимого формата → `400`.
- **E2E API**: загрузка фото при создании рецепта, проверка `hasPhoto` в ответе; удаление фото.
- **UI E2E**: карточка рецепта с фото отображает `<img>` вместо заглушки; кнопки управления фото видны автору.
