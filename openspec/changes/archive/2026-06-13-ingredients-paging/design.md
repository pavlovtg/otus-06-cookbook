# ingredients-paging

## Контекст

Текущий `GET /api/v1/ingredients` возвращает плоский массив всех ингредиентов без ограничений. Репозиторий стримит через `IAsyncEnumerable<Ingredient>`, контроллер буферизует в `List<IngredientDto>` и отдаёт весь результат. Поле `title` в фильтре не валидируется по длине на уровне API.

PRD требует пагинацию списков с размером страницы по умолчанию 100. Правила пагинации зафиксированы в стандарте REST API Design.

## Цели / Не-цели

**Цели:**

- Добавить пагинацию `GET /api/v1/ingredients` (page/pageSize, PagedResult).
- Ограничить `pageSize` максимумом 1000 (clamp, не ошибка).
- Валидировать `page` и `pageSize` как положительные числа (≥ 1); иначе `400`.
- Ограничить параметр `title` максимальной длиной 200 символов; иначе `400`.
- Применить ограничение `title` на всех уровнях: API, application, UI.

**Не-цели:**

- Пагинация других эндпоинтов (рецепты — отдельная задача).
- Курсорная пагинация.
- Изменение логики фильтрации по категории.

## Решения

### 1. Структура PagedResult

Новый DTO `PagedResult<T>` с полями `items`, `total`, `page`, `pageSize` возвращается контроллером вместо массива. Это breaking change контракта.

Альтернатива — заголовки `X-Total-Count` — отклонена: тело ответа самодостаточно и проще для клиентов.

### 2. Валидация параметров пагинации и title

Clamp `pageSize` до 1000 и валидация (`page ≥ 1`, `pageSize ≥ 1`) выполняются в контроллере до вызова сервиса — это граница API, не доменная логика.

Валидация длины `title` (≤ 200) выполняется в контроллере; константа берётся из `IngredientConstraints.TitleMaxLength`.

### 3. Слой Repository: Skip/Take + Count

`IIngredientRepository.GetIngredientsAsync` возвращает `Task<(IReadOnlyList<Ingredient> Items, int Total)>` вместо `IAsyncEnumerable<Ingredient>`. Репозиторий выполняет два запроса: `CountAsync` и `Skip/Take` — оба через один DbContext. Это соответствует исключению 3 из AR-0020 (пагинация с необходимостью вернуть `total`).

### 4. UI: пагинатор и ограничение поля поиска

Страница `/ingredients` получает `page` из `searchParams`, передаёт в BFF, отображает пагинатор. Поле поиска получает атрибут `maxLength={200}`.

## Риски / Компромиссы

- **BREAKING change контракта** → BFF и тесты обновляются в рамках этого же изменения.
- **Два SQL-запроса на список** → приемлемо для учебного проекта; при необходимости можно оптимизировать через `COUNT(*) OVER()`.
