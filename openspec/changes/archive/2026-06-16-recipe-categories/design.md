# recipe-categories

## Контекст

Сущность `Category` и её CRUD уже реализованы. Рецепт (`Recipe`) пока не связан с категориями — ни в доменной модели, ни в БД, ни в API, ни в UI. Изменение затрагивает все слои: домен, репозиторий, API-контракт, BFF, UI-форму и карточки рецептов.

Ключевые ограничения из действующих ADR и AR:

- Гексагональная архитектура (ADR-0013): доменная логика в `Domain/`, инфраструктура — в адаптерах.
- DDD (ADR-0012, AR-0007): связь рецепт↔категории моделируется внутри агрегата `Recipe`.
- Database-per-Service (ADR-0025, AR-0017): таблица `recipe_categories` живёт в БД cookbook-сервиса.
- Contract-First (ADR-0023, AR-0015): OpenAPI-контракт обновляется до реализации.
- EF Core (ADR-0014): конфигурация через `IEntityTypeConfiguration`, миграция через `MigrateDatabaseAsync`.

UI-стиль: `docs/design/mockup/styles.css` — единственный источник CSS-токенов. Компонент выбора категорий — `.tag-input` + `.chip` + `.autocomplete` (уже есть в mockup). Теги категорий в карточке — `.tags` + `.tag`. Новые компоненты добавляются в `docs/design/storybook/src/`.

## Цели / Не-цели

**Цели:**

- Добавить связь «рецепт ↔ категории» (many-to-many) в доменную модель и БД.
- Расширить API рецептов: принимать `categoryIds` (обязательное поле) и возвращать `categoryIds`.
- Добавить выбор категорий в форму создания/редактирования рецепта (tag-input с автодополнением).
- Отображать категории как теги в карточке и на детальной странице рецепта.
- Обновить seed-данные: назначить категории существующим рецептам.

**Не-цели:**

- Фильтрация рецептов по категориям (отдельная фича).
- Изменение CRUD категорий.
- Поддержка нескольких категорий одного типа на одном рецепте.

## Решения

### 1. Хранение связи: join-таблица `recipe_categories`

Связь many-to-many реализуется через join-таблицу `recipe_categories (recipe_id, category_id)` без отдельной сущности-агрегата. Альтернатива — хранить `category_ids` как массив в строке рецепта — отклонена: нарушает нормализацию и усложняет запросы.

### 2. Категории внутри агрегата `Recipe`

`Recipe` хранит `IReadOnlyList<CategoryId>`. Валидация «один тип — одна категория» выполняется в `Recipe.Create` / `Recipe.Update` через проверку дубликатов типов. Для проверки типа Application-сервис передаёт словарь `CategoryId → CategoryType` в доменный метод — домен не зависит от репозитория.

### 3. DTO: только идентификаторы категорий

Оба DTO рецептов возвращают только идентификаторы:

- `RecipeShortDto.categoryIds: uuid[]` — список рецептов
- `RecipeDto.categoryIds: uuid[]` — детальная страница и форма редактирования

Фронтенд загружает полный справочник категорий один раз (`GET /api/v1/categories`) параллельно с запросом рецептов и строит lookup-словарь `Map<id, CategoryDto>`. Это исключает JOIN при каждом запросе списка и позволяет переиспользовать справочник для формы редактирования.

### 4. `categoryIds` — обязательное поле в запросе

`RecipeRequest.categoryIds: uuid[]` — обязательное поле (может быть пустым массивом). Контракт обновляется в `docs/contracts/cookbook/recipes.yaml` до реализации (AR-0015).

### 5. Frontend: параллельная загрузка + инвалидация кеша

На странице списка рецептов: `Promise.all([getRecipes(...), getCategories()])`. Справочник категорий кешируется в React Query. При любой мутации каталога категорий (создание, редактирование, удаление) — инвалидация кеша `['categories']` через `queryClient.invalidateQueries`. Компонент выбора категорий в форме использует тот же кеш — дополнительный запрос не нужен.

## Риски / Компромиссы

- [Staleness] Если кеш категорий не инвалидирован вовремя — карточки покажут устаревшие теги. Митигация: инвалидация `['categories']` при каждой мутации каталога категорий.
- [Миграция] Новая таблица `recipe_categories` добавляется без данных; seed-данные назначаются через `CookbookSeeder` идемпотентно.
- [Валидация типов в домене] Домен получает типы категорий от Application — это допустимо: Application передаёт нужные данные, домен не зависит от репозитория.

## Тесты

| Уровень | Что проверяем |
|---------|--------------|
| Unit (домен) | `Recipe.Create` / `Recipe.Update` отклоняет две категории одного типа; принимает пустой список; заменяет категорию при совпадении типа |
| Integration (репозиторий) | Сохранение и загрузка `recipe_categories` через EF Core; идемпотентность seed-данных |
| Microservice (API) | `POST /api/v1/recipes` с `categoryIds` → `201` с корректными `categoryIds` в ответе; `PUT /api/v1/recipes/{id}` обновляет категории; `GET /api/v1/recipes/{id}` возвращает `categoryIds`; несуществующий `categoryId` → `422` |
| UI (Storybook) | `RecipeCard` отображает теги категорий; `RecipeForm` — tag-input добавляет/удаляет чипы, заменяет при совпадении типа |
| E2E | Создать рецепт с категориями → категории отображаются в карточке и на детальной странице; редактировать рецепт → категории обновляются |

## План миграции

1. Обновить OpenAPI-контракт (`categoryIds` в запросе и ответе).
2. Добавить EF-миграцию (join-таблица `recipe_categories`).
3. Реализовать изменения домена → Application → репозиторий → контроллер.
4. Обновить seed-данные.
5. Обновить Zod-схемы и BFF (параллельная загрузка категорий, инвалидация кеша).
6. Обновить UI-компоненты (tag-input в форме, теги в карточке).
7. Написать тесты.

Откат: миграция обратима (`Down` удаляет таблицу); фронтенд деградирует gracefully (`categoryIds` пустой массив → теги не отображаются).
