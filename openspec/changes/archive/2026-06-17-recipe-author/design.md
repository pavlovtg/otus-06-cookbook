# recipe-author

## Контекст

Рецепты хранятся без признака видимости и без ссылки на автора. Текущие схемы `RecipeShortDto` и `RecipeDto` не содержат полей `isPublic` и `authorName`. Форма создания/редактирования не предоставляет выбор видимости. Все рецепты доступны всем пользователям без ограничений.

## Цели / Не-цели

**Цели:**

- Добавить поля `is_public` и `author_id` в доменную модель и таблицу рецептов.
- Фильтровать приватные рецепты по правам: видны только автору и администратору.
- Отображать имя автора на карточке и детальной странице.
- Показывать тег «Приватный» на карточке и детальной странице приватного рецепта.
- Обновить OpenAPI-контракт и seed-данные.

**Не-цели:**

- Смена автора рецепта.
- Управление видимостью через отдельный эндпоинт.
- Уведомления при изменении видимости.

## Решения

### `author_id` — поле домена и БД, не выставляется в API-контракт

Поле `author_id` (uuid, nullable) хранится в таблице `recipes` и доменной модели. Используется для:

- фильтрации приватных рецептов (видны только автору);
- проверки прав на редактирование и удаление.

В API-контракт `authorId` не выставляется — клиенту достаточно `authorName` для отображения. Seed-рецепты могут не иметь автора (`author_id = null`).

### Поле `isPublic` — обязательное в запросе, без дефолта на сервере

Поле `isPublic` обязательно в `RecipeRequest`. Сервер не подставляет дефолт — клиент явно передаёт значение. Это исключает неявное поведение при создании рецепта.

Альтернатива: дефолт `true` на сервере — отклонена, так как скрывает намерение пользователя.

### `authorName` денормализуется в ответе API

`RecipeShortDto` и `RecipeDto` включают `authorName` (string, nullable) — отображаемое имя автора. Вычисляется через JOIN с таблицей пользователей при формировании ответа. Nullable — для рецептов без привязанного автора.

Альтернатива: возвращать `authorId` и разрешать имя на фронте — отклонена, так как требует дополнительного запроса и усложняет BFF.

### Чужой приватный рецепт возвращает `403 Forbidden`

`GET /api/v1/recipes/{id}` возвращает `403` при попытке получить чужой приватный рецепт. Факт существования рецепта не скрывается — это упрощает отладку и соответствует семантике «доступ запрещён».

Альтернатива: `404` — отклонена по решению команды (см. specs).

### Seed-данные: все существующие рецепты — публичные

При миграции все существующие рецепты получают `is_public = true`. Новые seed-рецепты явно задают значение.

## Риски / Компромиссы

- **Nullable `author_id`**: seed-рецепты могут не иметь автора — `authorName` в ответе будет `null`; UI должен обрабатывать отсутствие значения.
- **Денормализация `authorName`**: при смене `display_name` пользователя ответы API вернут актуальное имя (JOIN при запросе) — это ожидаемое поведение для MVP.
- **Миграция**: добавление NOT NULL поля `is_public` требует дефолтного значения в миграции (`true`) для существующих строк → безопасно, все старые рецепты становятся публичными.
