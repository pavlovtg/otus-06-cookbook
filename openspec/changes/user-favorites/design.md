# user-favorites

## Контекст

Приложение уже имеет авторизацию (JWT, таблица `users`), список рецептов с пагинацией и поиском. Избранное — новая связь «пользователь ↔ рецепт», хранимая на сервере. Архитектура backend — гексагональная (Ports & Adapters) + DDD; frontend — Next.js App Router + BFF.

## Цели / Не-цели

**Цели:**

- Хранить избранное пользователя на сервере (таблица `user_favorites`).
- Предоставить API: добавить/удалить рецепт из избранного; получить список рецептов с фильтром по избранному.
- Отображать признак `isFavorite` на карточке и детальной странице рецепта.
- Переключатель «Все рецепты» / «Избранное» в сайдбаре страницы рецептов.
- Поиск работает внутри подмножества избранных при активном режиме «Избранное».
- Seed-данные: несколько записей избранного для `ivlev@cookbook.local` и `renat@cookbook.local`.

**Не-цели:**

- Публичный список избранного (виден только владельцу).
- Сортировка избранного по дате добавления.
- Использование избранного в планировщике меню (отдельная фича).

## Решения

### 1. Таблица `user_favorites` — простая связь без агрегата

Избранное — слабая связь без собственного жизненного цикла: `(user_id, recipe_id)` с уникальным ограничением. Отдельный агрегат не нужен — операции выполняются через `RecipeRepository` (тот же bounded context).

Каскадное удаление: при удалении пользователя или рецепта все связанные записи `user_favorites` удаляются автоматически через `ON DELETE CASCADE` на обоих внешних ключах.

Альтернатива: отдельный агрегат `Favorite` — избыточно для простой связи без бизнес-правил.

### 2. API: POST/DELETE `/api/v1/recipes/{id}/favorites`

Toggle реализован двумя отдельными эндпоинтами (POST = добавить, DELETE = убрать), а не одним toggle-эндпоинтом. Это идемпотентно: повторный POST не создаёт дубликат (upsert), повторный DELETE возвращает 204 без ошибки.

Альтернатива: единый `PUT /favorites` с телом `{isFavorite: bool}` — менее REST-идиоматично.

### 3. Поле `isFavorite` в `RecipeShortDto` — контекстное

`isFavorite` возвращается только для авторизованных запросов (из JWT `sub`). Для анонимных — `null` или поле отсутствует. Это позволяет не менять контракт для неавторизованных клиентов.

### 4. Фильтр `favorites=true` в `GET /api/v1/recipes`

Добавляется булев параметр `favorites`. При `favorites=true` без JWT — `401`. Поиск (`q`) применяется поверх отфильтрованного подмножества.

### 5. Frontend: серверный BFF читает `isFavorite`, клиентский компонент делает toggle

Список рецептов — Server Component, получает `isFavorite` из BFF. Кнопка «В избранное» — Client Component (нужен `onClick`). Toggle вызывает Route Handler `/api/favorites/[id]`, который проксирует к backend.

## Тесты

- **Unit (backend)**: `RecipeService` — фильтрация по `favorites=true`, поле `isFavorite` в результате.
- **Integration (backend)**: репозиторий — `AddFavorite`, `RemoveFavorite`, идемпотентность, каскадное удаление при удалении рецепта/пользователя.
- **Microservice (backend)**: `POST /api/v1/recipes/{id}/favorites` (авторизован / не авторизован), `DELETE /api/v1/recipes/{id}/favorites`, `GET /api/v1/recipes?favorites=true`.
- **Unit (frontend)**: Zod-схема `RecipeShortDtoSchema` — поле `isFavorite`; BFF `favorites.ts` — mock fetch.
- **E2E API**: добавить/удалить избранное, получить список с `favorites=true`.
- **UI E2E**: переключение режима «Избранное» в сайдбаре, иконка на карточке.

## UI-дизайн

Реализация следует макету `docs/design/mockup/index.html` и токенам `docs/design/mockup/styles.css`:

- Иконка-сердечко на карточке рецепта: позиция `position: absolute; top: 10px; right: 10px` поверх фото, класс `btn-icon photo-fav`; активное состояние — `is-on` (цвет `--accent-pink`).
- Иконка на детальной странице: кнопка в `.detail-actions`, класс `btn btn-primary btn-sm` (активное) / `btn btn-ghost btn-sm` (неактивное).
- Переключатель в сайдбаре: элементы `.aside-item` с `data-mode="all"` и `data-mode="favorites"` в блоке `.aside-block`; активный — класс `is-active`.
- Пустое состояние избранного: компонент `.state` с текстом «В избранном пусто» / «Сохраните любимое».
- Компоненты разрабатываются в Storybook (`docs/design/storybook/`) перед интеграцией в `apps/web`.

## Риски / Компромиссы

- **N+1 при `isFavorite`** → Mitigation: один JOIN `user_favorites` при запросе списка рецептов (не отдельный запрос на каждый рецепт).
- **Race condition при одновременном toggle** → Mitigation: уникальный индекс на `(user_id, recipe_id)` + upsert/ignore на уровне БД.
- **Оптимистичный UI** → не реализуется в MVP; кнопка блокируется до ответа сервера.
