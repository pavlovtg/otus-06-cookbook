# recipes-search

## Контекст

Приложение имеет список рецептов с пагинацией, но без поиска и сортировки. Backend реализован на .NET 10 / EF Core с PostgreSQL; frontend — Next.js с серверным BFF. Поиск затрагивает оба слоя: backend добавляет фильтрацию в запрос к БД, frontend добавляет UI-элементы и сохраняет состояние в URL.

## Цели / Не-цели

**Цели:**

- Поиск по строке (название, описание, категории, ингредиенты; AND по словам) через параметр `q`.
- Сортировка по названию (`title_asc` / `title_desc`) через параметр `sort`.
- Автодополнение из категорий и ингредиентов на стороне frontend.
- Состояние поиска и сортировки в URL (`?q=...&sort=...&page=...`).

**Не-цели:**

- Полнотекстовый поиск PostgreSQL (FTS, tsvector) — не нужен на текущем масштабе.
- Поиск по полю `instructions`.
- Сортировка по рейтингу — отдельная задача.
- Отдельная страница результатов поиска.

## Решения

### Фильтрация на стороне backend

Поиск реализуется через LINQ-запрос с `EF.Functions.ILike` (регистронезависимый LIKE в PostgreSQL) по каждому слову запроса.
Каждое слово применяется как отдельный `.Where(...)` — это даёт AND-семантику.
Поиск по категориям и ингредиентам требует JOIN через навигационные свойства, которые уже загружаются в `GetRecipesPagedAsync`.

Альтернатива — PostgreSQL FTS — избыточна: нет требований к релевантности, стеммингу или весам.

### Параметр sort

Значения `title_asc` и `title_desc` — строковый enum в контроллере.
Сортировка применяется через `.OrderBy` / `.OrderByDescending` по полю `Title`.

### Автодополнение на frontend

Автодополнение реализуется на стороне клиента: при вводе ≥ 2 символов последнего слова фильтруются уже загруженные списки категорий и ингредиентов (они загружаются при открытии страницы).
Отдельного API-эндпоинта для автодополнения не требуется.

### URL-state

Параметры `q`, `sort`, `page` хранятся в URL через `searchParams` Next.js.
Страница является серверным компонентом (RSC), поэтому поиск и сортировка выполняются на сервере при каждом запросе.

## Риски / Компромиссы

- [Производительность при большом числе рецептов] → LIKE-запросы без индекса замедляются при тысячах записей. Для MVP (25+ рецептов) приемлемо; при росте — добавить GIN-индекс или FTS.
- [JOIN по категориям и ингредиентам] → Увеличивает сложность запроса. Митигация: `AsNoTracking`, запрос выполняется один раз с Include.
- [Автодополнение из памяти] → При большом числе категорий/ингредиентов список может быть большим. Для MVP (50+ ингредиентов, ~60 категорий) приемлемо.
